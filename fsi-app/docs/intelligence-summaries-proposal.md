## Two paths

The user asked me to evaluate either retire (Path A) or regenerate (Path B). My recommendation: **Path A**.

---

### Path A â€” Retire `SectorSynopsisView` entirely; replace with dynamic per-sector via `topic_tags` + workspace profile

**What changes:**

1. **DB:** drop `intelligence_summaries` table (or `TRUNCATE` for one phase if conservative). Drop the two indexes and any FK references. New migration: `024_drop_intelligence_summaries.sql`.

2. **Server-side fetch:** remove `fetchAllSynopses()` from `src/lib/supabase-server.ts`. Remove `intelligence_changes` and `sector_contexts` from the same `Promise.all` if they're no longer consumed elsewhere (verify first â€” they likely are).

3. **Stores:** remove `StoredSynopsis` type from `src/stores/resourceStore.ts`. Remove synopsis state and `setSynopses` action. Update any selectors that read `state.synopses`.

4. **UI replacement** â€” `SectorSynopsisView` becomes a thin shim around `IntelligenceBrief` + `IntelligenceMetadataStrip`:

   ```tsx
   export function SectorSynopsisView({ itemId, fullBrief, fallbackWhatIsIt, fallbackWhyMatters, fallbackKeyData }) {
     // No more synopsis state. Per-sector relevance is derived from
     // the workspace profile + item tags at render time.
     if (fullBrief) return (
       <>
         <IntelligenceMetadataStrip itemId={itemId} />
         <IntelligenceBrief markdown={fullBrief} />
       </>
     );
     // Fallback for items not yet regenerated under the new contract
     return <FallbackBriefView fallbackWhatIsIt={fallbackWhatIsIt} ... />;
   }
   ```

5. **Workspace-aware sector relevance:** the new model surfaces sector-relevance through tag intersection, not pre-baked text:
   - Workspace profile carries `active_sectors[]`
   - Item carries `compliance_object_tags[]` (which supply-chain roles) + `operational_scenario_tags[]` (which scenarios)
   - At render time: if any of the workspace's active sectors map to a compliance_object_tag the item carries, surface a "This applies to your <sector>" callout above the brief; otherwise silence
   - Per-sector specialized text isn't pre-generated â€” it's the brief itself, which is already workspace-anchored per the Workspace-Anchored Rule in SKILL.md

6. **Cleanup:** delete `add-building-standards.mjs` and any other one-shot `intelligence_summaries` writers per Rule 11 (deprecation = deletion).

**Cost:**
- DB: $0 (TRUNCATE / DROP)
- Code: ~1 day of UI work (4â€“6 commits across server-fetch / store / SectorSynopsisView / cleanup)
- Ongoing: $0 (no parallel content pipeline to maintain)

**Trade-offs:**
- **+** Single source of truth (`intelligence_items.full_brief` + the 12 metadata fields)
- **+** No drift between brief content and per-sector summaries
- **+** No ongoing regeneration cost
- **+** Workspace-anchoring is already enforced at the brief level by the agent contract â€” the sector layer was redundant
- **âˆ’** Loss of the per-sector "translated" headline. Some users may have valued the punchier sector-specific summary. Mitigation: the 14-section regulatory_fact_document already opens with a "What This Regulation Is and Why It Applies to the Workspace" section that fills the same need.

## Recommendation: Path A

Three reasons:

1. **Cost asymmetry.** Path A is $0 + 1 day of UI work. Path B is $60+ per regeneration cycle plus ongoing maintenance plus a synchronization problem.

2. **Single source of truth.** The 13-field intelligence_items contract already carries full content (`full_brief`) + workspace-anchoring (Workspace-Anchored Rule) + role-targeting (`compliance_object_tags`) + scenario-targeting (`operational_scenario_tags`). A separate per-sector layer is redundant and creates a two-source-of-truth drift problem.

3. **The new contract subsumes the old need.** What `SectorSynopsisView` did â€” give workspace operators a sector-specific lens on the regulation â€” is now done by:
   - The brief's Section 2 ("What This Regulation Is and Why It Applies to the Workspace") for regulatory_fact_documents
   - The format-specific sections that already cover competitive / client-conversation / action lenses for non-regulation items
   - The new `compliance_object_tags` and `operational_scenario_tags` that let the UI filter "regulations that apply to my sector" without per-pair pre-generation

The intersection-detection feature shipped this phase already operates on the same tag axes â€” if those tags are good enough for cross-regulation matching, they're good enough for per-workspace-sector relevance.
