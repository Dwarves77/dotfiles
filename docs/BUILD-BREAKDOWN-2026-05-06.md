## 1. What this is

**Caro's Ledge** is a freight-sustainability intelligence platform for high-value cargo forwarders. Not a regulation database ŌĆö a **source-monitoring system** covering 7 intelligence domains:

1. Regulatory & Legislative
2. Energy & Technology Innovation
3. Regional Operations Intelligence
4. Geopolitical & Market Signals
5. Source Intelligence (the meta-layer that tracks the registry itself)
6. Warehouse & Facility Optimization
7. University & Research Pipeline

The architecture distinguishes two layers:

- **Layer 1 ŌĆö Sources**: public portals where authoritative content lives (EUR-Lex, Federal Register, IMO, EPA, etc.). 563 total ┬Ę 501 active.
- **Layer 2 ŌĆö Intelligence Items**: specific regulations, technology profiles, market signals, operations profiles, and research findings that live *inside* sources. 194 items ┬Ę 184 active. Each is generated by an AI agent and stored as a structured brief.

The platform monitors sources ŌåÆ sources produce intelligence items ŌåÆ items are filtered by the user's workspace profile (sectors + jurisdictions + transport modes) for relevance.

### Operating principle

> **Creative intelligence, accurate grounding.** Be creative about what to find, conservative about what to claim. If a fact can't be grounded in a verifiable source, omit it. Surface gaps explicitly. Flag new sources as provisional. Document non-obvious intersections.

This is the platform's differentiator: generic LLMs are creative but unreliable, conservative compliance tools are reliable but limited. Caro's Ledge does both ŌĆö within enforced integrity rules and a 7-tier source trust framework.

## 5. Agent runtime

### Format-selected single-brief regeneration

`POST /api/agent/run` takes a `sourceUrl`, fetches the `intelligence_items` row matching that URL, and regenerates ONE brief in the format dictated by `item_type`:

| item_type | Format | Sections |
|---|---|---|
| regulation, directive, standard, guidance, framework | regulatory_fact_document | 14 (8 conditional) |
| technology, innovation, tool | technology_profile | 8 |
| regional_data | operations_profile | 8 |
| market_signal, initiative | market_signal_brief | 8 |
| research_finding | research_summary | 6 |

The agent emits 13 fields per regeneration:
- `full_brief` markdown body
- 12 YAML metadata fields including `severity`, `priority`, `urgency_tier`, `format_type`, `topic_tags`, `compliance_object_tags`, `operational_scenario_tags`, `related_items`, `intersection_summary`

### Contract version + parser

Operative source of truth: `fsi-app/src/lib/agent/system-prompt.ts` (~18.6 KB).
Reference + contract: `fsi-app/.claude/skills/environmental-policy-and-innovation/SKILL.md` (~42 KB).
Both content-aligned; YAML emits `regeneration_skill_version: "2026-04-29"`.

Parser: 3-tier YAML fallback in `parse-output.ts`. Extracts the 12-field YAML frontmatter from the trailing `---` block and validates against closed-vocabulary enums. **Does NOT enforce section names or order** ŌĆö agent can drift sections freely without a runtime check. 4 of 7 audited briefs are out of spec section order ŌĆö invisible to the parser.

### Integrity rule

> No invented facts, no hallucinated content, no plausible-sounding generic filler. When source coverage is thin, sections are honestly omitted. When canonical sources can't be found, the gap is flagged. All synthesis is grounded in component sources cited inline.

When the agent self-flags ("integrity rule" phrase in markdown), the trigger from mig 035 + retune in mig 044 routes the row to the admin queue. Currently 1 unresolved (post-retune, was 57 pre-retune).

### Workspace-anchored rule

Every brief is generated with workspace context (sectors, jurisdictions, modes). Output never names companies/personal/products ŌĆö generic-but-in-scope language driven by workspace profile.

### Cross-format lens

Every brief serves four lenses: substantive (legal/regulatory) ┬Ę competitive ┬Ę client-conversation ┬Ę action.

### Intersection detection

Migration 020 + 021/023 added `operational_scenario_tags`, `compliance_object_tags`, `related_items`, `intersection_summary` to every item, with GIN indexes. RPC `detect_intersections(min_strength, max_results)` joins the items table against itself, scores pairs by shared tags + explicit links + both-high-priority, returns ranked pairs.

UI: "Intersections" sub-tab in SourceHealthDashboard. Top hits include CIIŌåöIMO GHG (strength 15), Crude/Jet FuelŌåöReFuelEU (12), UK MEESŌåöNYC Local Law 97 (10).

### Source-discovery pipeline

1. **Discovery**: agent runs emit a `## New Sources Identified` markdown table after the brief body. Citation extractor (`/api/agent/run` Step 8a-8b) parses it and writes `source_citations` rows (existing source match) or upserts `provisional_sources` (new URL).
2. **Verification (W2.F)**: `tier1-population-runner.mjs` plus the W2.F pipeline classify candidates via Haiku scoring on relevance + freight; route to H (auto-active source) / M (queue to provisional) / L (rejected, audit-only).
3. **Promotion**: `/api/admin/sources/promote` with approve/reject/defer decisions. Approve writes a new `sources` row + a `source_trust_events` audit row. UI: ProvisionalReviewCard with cached AI rationale.
4. **Trust scoring**: Bayesian-prior blend (`tierPrior + earned`, weighted by signal density). Monthly recompute via `.github/workflows/trust-recompute.yml` (cron 03:00 UTC on the 1st).
5. **Spot-check**: pilot of 20 sources caught 15% FP rate at 70/50 thresholds. PR #26 tightened to **75/55** + demoted 3 named (DPNR/MDE/VDOT). `spot-check-all-h-tier.mjs` ready to run on remaining ~61 H-tier (~$0.06 Haiku, **not yet dispatched**). Recurring monthly cron `spot-check-monthly.yml` wired (1st of month, fails CI if FP rate > 5%).

### Permitted / non-negotiable rules

- DO NOT change `/api/agent/run` to per-sector calls. Format-selected single-brief is the architecture.
- DO NOT make live Claude API calls outside the 7 routes above.
- DO NOT process provisional sources. Every API call, scrape job, AI pipeline, embedding generation, health check, search indexing task MUST gate on `WHERE status='active' AND admin_only=false`.
