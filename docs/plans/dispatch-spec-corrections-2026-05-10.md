# Dispatch spec corrections — 2026-05-10

Three errors in dispatch specs surfaced across the source-classification metadata workstream. Captured here so future dispatches stop tripping over them.

## 1. Env path is `.env.local`, not `.env`

Multiple dispatches reference `.env` for credentials. Live repo uses `.env.local` (matches Next.js convention). Scripts that call `process.loadEnvFile(".env")` silently load nothing and then fail at `createClient(...)` with "Missing NEXT_PUBLIC_SUPABASE_URL" or similar.

**Correction:** specs and scripts must use `.env.local`. Verified locations:
- `C:/Users/jason/dotfiles/fsi-app/.env.local` — service role key, anon key, URL, DB password
- Pooler URL and project ref are read from `supabase/.temp/pooler-url` and `supabase/.temp/project-ref` for direct pg connections.

## 2. supabase-js cannot wrap multiple PostgREST calls in a transaction

Dispatches keep asking for "wrap the batch in a transaction." This is impossible with supabase-js because each call is an independent HTTP request to PostgREST; there is no client-side transaction boundary that spans multiple `.update()` / `.insert()` calls.

**Established pattern in this repo:** per-row UPDATEs with halt-on-first-error and an idempotent WHERE guard. See `scripts/backfill-classify-batch.mjs` — first failure breaks the loop, no further writes attempted, guard makes re-runs of the same batch a no-op for already-written rows.

**If true atomicity is required:** open a direct pg connection (the pattern in `supabase/seed/apply-pending.mjs`), wrap in `BEGIN ... COMMIT`, and bypass PostgREST entirely. This is the path used for migration applies. Acceptable for write-heavy backfills too if the atomicity requirement is genuine.

## 3. Schema audit must run against the live DB before any dispatch claims a column exists

Migration files alone are insufficient evidence that a column exists with the expected type. Concrete failure mode in this codebase:

- Migration 063 included `ADD COLUMN IF NOT EXISTS tier TEXT NULL`.
- An older `tier` column already existed on `public.sources` as `INTEGER`.
- `IF NOT EXISTS` made the new ADD a silent no-op — no error, no type change.
- Downstream code that wrote `tier = 'T1'` (string) failed with type errors, while code that wrote `tier = 1` (integer) succeeded with the wrong semantics.

**Correction:** before any dispatch claims a column is present at a given type, query `information_schema.columns` against the live DB. The `apply-pending.mjs` connection pattern works for this. A 5-line probe script before the dispatch is cheaper than a half-applied backfill.

This same audit caught the `classification_confidence` / `classification_rationale` gap — those columns did not exist on `public.sources` despite being assumed by an earlier dispatch. Fixed in migration 067 (2026-05-10).

## 4. Log filename date stamps use America/New_York local date, not UTC

Log filenames generated by classification scripts use America/New_York local date, not UTC. After ~20:00 ET the UTC date rolls over to the next calendar day, which produced confusing artifacts like `source-classification-batch-1-log-2026-05-11.md` written during a 2026-05-10 ET session.

Implemented in:
- `scripts/backfill-classify-batch.mjs`
- `scripts/backfill-classify-metadata-batch-1.mjs`

Both scripts now compute `LOG_DATE` via `Intl.DateTimeFormat('en-CA', { timeZone: 'America/New_York', year:'numeric', month:'2-digit', day:'2-digit' }).format(new Date())`. `en-CA` produces ISO order natively, so no reassembly is required.

Existing files were not renamed; forward consistency only. Operator policy 2026-05-10.
